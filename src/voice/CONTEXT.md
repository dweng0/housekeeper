# Voice Context

Handles always-on speech transcription from distributed Voice Nodes, detects Directed Questions, and dispatches classified intent to the Automation and Memory contexts. Routes TTS responses back to the originating Voice Node (or the default output node).

## Language

**Voice Node**:
A device registered with the server that provides audio input, output, or both. Has a label, a location, declared capabilities, and a transport. Two transports exist: WebSocket (Pi-based devices) and Cast (Google Cast devices, e.g. Nest Mini).
_Avoid_: edge device, smart speaker, microphone, client

**Transport**:
The delivery mechanism a Voice Node uses. `"websocket"` — Pi device connects to the server over WebSocket and receives raw PCM audio pushed to it. `"cast"` — Google Cast device discovered via mDNS; server sends it a URL to fetch and play.
_Avoid_: type, protocol, connection type

**WebSocket Node**:
A Voice Node with `transport: "websocket"`. Pi-based. Self-registers by connecting to the server WebSocket. Can declare `mic`, `speaker`, or both capabilities.
_Avoid_: Pi node, edge device

**Cast Node**:
A Voice Node with `transport: "cast"`. A Google Cast device (e.g. Nest Mini) on the local network. Discovered automatically via mDNS — never self-registers. Always declares `["speaker"]` capability only. Identified by the Cast device UUID broadcast via mDNS.
_Avoid_: smart speaker, Nest Mini, Chromecast node

**Capabilities**:
The audio roles a Voice Node declares: `mic` (input), `speaker` (output), or both. Cast Nodes always declare `["speaker"]` only. WebSocket Nodes declare `["mic"]` or `["mic", "speaker"]`.
_Avoid_: type, mode, features

**Location**:
A free-text string describing where in the house a Voice Node is placed (e.g. `"hallway"`, `"upstairs bedroom"`). Used by the LLM to resolve implicit spatial references in Directed Questions.
_Avoid_: room, zone, area

**Default Output Node**:
The globally configured Voice Node used as fallback for TTS responses when the originating Voice Node has no speaker capability. Configured system-wide via the dashboard. If offline, the response is logged and dropped.
_Avoid_: fallback speaker, default device

**Directed Question**:
An utterance that contains the System Name and is interpreted as addressed to the system. The name may appear anywhere in the sentence.
_Avoid_: wake word, command, voice command

**System Name**:
The configurable name the household uses to address the AI (e.g. "housekeeper"). Detected by text scan of the Listening Window.
_Avoid_: wake word, hotword

**Utterance**:
A single continuous spoken sentence or phrase, bounded by a Speech Boundary. Transcribed on the Voice Node and sent as text to the server.
_Avoid_: command, speech, input

**Listening Window**:
A per-Voice-Node rolling transcript of recent Utterances. When the System Name is detected within it, the full window is dispatched for classification.
_Avoid_: context window, audio buffer, transcript buffer

**Speech Boundary**:
The point of detected silence (via VAD debounce, ~700ms–1s) that closes an Utterance on the Voice Node. Handled on the Voice Node, not the server.
_Avoid_: end of speech, silence detection, cutoff

**Confirmation Response**:
The natural-language text the LLM generates alongside a Classified Intent, spoken back to the resident via TTS after the action is dispatched. Styled to match the system persona (e.g. *"Done — the hallway light is now on."*). Generated in the same LLM call as classification; not a separate round trip.
_Avoid_: reply, message, feedback string

**Classified Intent**:
The structured output of the Intent Classifier for a Directed Question. Carries a `type` discriminator (`device-control`, `create-automation`, `set-resident`, `query`, `unknown`), type-specific fields, an optional `residentName` (present when the speaker identifies themselves in the same utterance as an action), an optional `response` (the Confirmation Response), an optional `hedgedResponse` (the Hedged Response), and an `intentConfidence` score (0–1).
_Avoid_: parsed command, NLU result

**Intent Confidence**:
A 0–1 float returned alongside a Classified Intent, representing the LLM's certainty that it correctly identified the intent from the utterance. Below the configured threshold (default `0.7`), the Hedged Response is spoken instead of the direct Confirmation Response. Applies to all intent types including `query`. `unknown` type ignores confidence — no speech is produced regardless.
_Avoid_: score, certainty, probability

**Hedged Response**:
The spoken variant of a Confirmation Response used when Intent Confidence is below the configured threshold. Styled to signal uncertainty while still acting (e.g. *"I think you're asking me to turn on the hallway light — done"*). Generated by the LLM in the same call as the direct Confirmation Response. Low-confidence intents always use live TTS for the Hedged Response — the Response Audio Cache is bypassed.
_Avoid_: uncertain response, fallback response, low-confidence reply

**Response Audio Cache**:
The on-disk store of pre-rendered audio files for known `{deviceLabel, command}` pairs. Keyed by `"deviceLabel:command"` for positive variants, plus a single shared `__not_found__` pool for the device-not-registered case. Index lives at `data/response-cache/index.json`. Built at startup (incremental — missing entries only) and when a device is added. Fully rebuilt on demand via the dashboard.
_Avoid_: TTS cache, audio cache, preload

**Pre-rendered Response**:
A set of Response Variants for a specific `{deviceLabel, command}` pair, stored in the Response Audio Cache. Covers the success outcome only — the not-found outcome is handled by the shared `__not_found__` pool. Generated by the LLM (text variants styled to the current persona) and rendered to WAV by piper at build time.
_Avoid_: cached response, preloaded audio

**Response Variant**:
One WAV file within a Pre-rendered Response or the `__not_found__` pool. Selected randomly at playback time to avoid repetition. Default count: 3 per entry (configurable via dashboard; higher counts trade disk space for more variation).
_Avoid_: audio file, take, sample

**Conversation Context**:
The active per-Voice-Node thread of exchanges that opens after a Directed Question receives a spoken response. While open, subsequent Utterances are treated as Follow-up Utterances and routed to the system without requiring the System Name. Scoped per Voice Node. Resets on: timeout (configurable, default 30s), a new Directed Question, or a Resident change (`set-resident` intent).
_Avoid_: context window, conversation session, dialogue state

**Follow-up Utterance**:
An Utterance that arrives while a Conversation Context is open and does not contain the System Name. Routed to the Intent Classifier with the full Conversation History attached. All intent types are valid — a Follow-up Utterance may control a device, create an Automation, or ask a query, resolved against the prior exchange.
_Avoid_: follow-up question, continuation, chained command

**Conversation History**:
The ordered sequence of prior user Utterances and system Confirmation Responses within an active Conversation Context. Passed to the Intent Classifier as a structured OpenAI message array (`role: user/assistant` turns). Bounded by a rolling token budget — oldest turns are dropped first when the budget is exceeded.
_Avoid_: chat history, message log, prior context

**Conversation Turn**:
One user Utterance + one system Confirmation Response pair within a Conversation Context. The unit of history in the Conversation History array.
_Avoid_: exchange, round, interaction

## Relationships

- A **WebSocket Node** sends zero or more **Utterances** to the server over WebSocket
- A **Cast Node** never sends Utterances — it is output-only
- A **Listening Window** is maintained per **WebSocket Node** on the server
- A **Directed Question** is one **Utterance** that references the **System Name**
- A **Directed Question** that receives a spoken response opens a **Conversation Context** on that **Voice Node**
- A **Follow-up Utterance** is routed like a **Directed Question** but carries **Conversation History**
- A **Conversation Context** is reset by timeout, a new **Directed Question**, or a **Resident** change
- A TTS response routes to the originating **Voice Node** if it has speaker capability; otherwise routes to the **Default Output Node**
- Multiple **Voice Nodes** of either transport may be connected simultaneously

## Registration

**WebSocket Nodes** self-register on connect, sending a handshake:
- `id` — stable identifier (e.g. hostname or UUID, persisted on device)
- `label` — human-readable name (e.g. `"hallway"`)
- `location` — free-text location string (e.g. `"downstairs hallway"`)
- `capabilities` — `["mic"]` or `["mic", "speaker"]`

**Cast Nodes** are discovered automatically via mDNS. The hub listens for Google Cast service announcements on the local network. Discovered devices surface in the dashboard as unconfirmed — the user assigns a label and location before the node becomes active. The Cast device UUID is used as `id` directly.

In both cases: if unconfirmed, the node surfaces in the dashboard. If already registered, it reconnects or reappears by ID.

## TTS Delivery

TTS audio is rendered by piper on the server (consistent voice across all nodes regardless of transport).

- **WebSocket Node**: rendered PCM buffer is pushed directly over the WebSocket connection.
- **Cast Node**: rendered buffer is written to a temp file, served over HTTP by the server, and the Cast device is instructed to fetch and play that URL via the Cast protocol (TLS, port 8009).

The hub owns transport-specific delivery. The TTS adapter is oblivious to transport.
