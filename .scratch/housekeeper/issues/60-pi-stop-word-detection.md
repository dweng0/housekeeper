Pi listens for stop-words during TTS playback and signals the server to interrupt. When user says "wait" or "stop" during device-control response, Pi detects it and sends a message back so the server can play a confirmation and offer to replay.

**Where does it touch?**

- Voice node audio processing loop (separate voicenode repo)
- WebSocket message sending back to server
- No changes to housekeeper server

**What's already there?**

- WebSocket already sends utterances to server
- Same connection used for stop-word signal (piggyback on existing WebSocket)

**How to verify?**

1. Hard-code 7 stop-words: `wait`, `stop`, `hold on`, `cancel`, `no`, `nope`, `never mind`
2. Build regex for whole-word matching: `\b(wait|stop|hold on|cancel|no|nope|never mind)\b` (case-insensitive)
3. On `tts_stream_start`: begin listening for stop-words
4. On match: send `{ "type": "stop_word", "keyword": "<matched>" }` immediately over WebSocket
5. On `tts_stream_end`: stop listening
6. On timeout (30s without `tts_stream_end`): stop listening anyway
7. Unit tests: verify regex matches each word in isolation, case-insensitivity, word boundary (no false positives on "desktop" for "stop")
8. Integration test: spin up mock server, send `tts_stream_start`, speak "wait", verify `{ "type": "stop_word", "keyword": "wait" }` arrives at server
