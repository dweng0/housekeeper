# Conversation Context: wire into VoiceAutomationService

## What to build

Wire `ConversationContext` and the updated `IntentClassifier` port into `VoiceAutomationService`. After any spoken TTS response, open (or extend) the Conversation Context on the originating node. When an utterance arrives without the System Name and a Conversation Context is open on that node, treat it as a Follow-up Utterance: classify it with the full Conversation History attached. Reset context on a new Directed Question or a `set-resident` intent.

End-to-end: a user can say "Hey Housekeeper, turn on the porch light", receive a response, then say "actually turn it off" or "but why?" without repeating the system name — and the system resolves it correctly against the prior exchange.

## Acceptance criteria

- [ ] After any spoken response, `ConversationContext` for that node is opened/extended with the new turn
- [ ] Follow-up Utterances (no System Name, open context) are routed to the classifier with `conversationHistory` populated
- [ ] New Directed Question on the same node resets the context before classifying
- [ ] `set-resident` intent resets the context
- [ ] Context timeout (no speech for N seconds) closes the context — subsequent utterances without System Name are ignored as normal ambient speech
- [ ] All intent types (`device-control`, `create-automation`, `query`, `set-resident`, `unknown`) valid as follow-up outcomes
- [ ] Integration tests cover: follow-up query, follow-up device-control, context reset by new directed question, context reset by resident change, context expiry

## Agent Brief

**Category:** enhancement
**Summary:** Wire `ConversationContext` and follow-up utterance routing into `VoiceAutomationService` and `ListeningWindow`.

**Current behavior:**
`ListeningWindow` fires `onDirectedQuestion` only when an utterance contains the system name; all other utterances are silently discarded. `VoiceAutomationService` has no `ConversationContext` per node and always classifies with no history.

**Desired behavior:**
After any spoken TTS response, the `ConversationContext` for the originating node is opened or extended with that turn. When a subsequent utterance arrives on the same node **without** the system name, and the `ConversationContext` for that node is still open, it is treated as a Follow-up Utterance: classified with `conversationHistory` populated from the context. If the context has expired or was never open, the utterance is ignored as ambient speech (existing behavior). A new Directed Question (system name present) resets the context before classifying. A `set-resident` intent outcome resets the context.

**Key design note — follow-up routing:**
`ListeningWindow` currently does not surface utterances without the system name at all. To route follow-ups, `ListeningWindowOptions` needs a second optional callback, e.g. `onAmbientUtterance?: (transcript: string) => void`, fired for utterances that do not contain the system name. `VoiceAutomationService` wires this callback and checks `ConversationContext.isOpen()` to decide whether to classify or discard.

**Key interfaces:**
- `ListeningWindowOptions` — add `onAmbientUtterance?: (transcript: string) => void`; fire it for every utterance that does not trigger `onDirectedQuestion`
- `VoiceAutomationService` — maintain a `Map<nodeId, ConversationContext>` (one context per node); pass `onAmbientUtterance` when constructing each `ListeningWindow`
- After every `speechOutput.speak(...)` call for a Directed Question response, call `context.addTurn(utterance, spokenText)` on the node's context
- On `onAmbientUtterance`: if `context.isOpen()`, call `classifier.classify({ utterance, ..., conversationHistory: context.getHistory(budget) })`, then handle the result identically to a Directed Question result; otherwise discard
- On `onDirectedQuestion`: call `context.reset()` before classifying
- On `set-resident` intent outcome: call `context.reset()`

**Acceptance criteria:**
- [ ] After spoken response, the node's `ConversationContext` contains the turn (utterance + spoken text)
- [ ] Follow-up utterance (no system name, open context) is classified with `conversationHistory` from the context
- [ ] Follow-up utterance with expired/absent context is discarded (no classification)
- [ ] New Directed Question resets context before classifying
- [ ] `set-resident` intent resets context
- [ ] Integration tests cover: follow-up query resolved correctly, follow-up device-control resolved correctly, context reset by new directed question, context reset by resident change, context expiry causes follow-up discard
- [ ] All existing `VoiceAutomationService` and `ListeningWindow` tests remain green
