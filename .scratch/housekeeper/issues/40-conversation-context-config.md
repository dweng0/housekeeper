**Category:** enhancement
**Summary:** Add `conversationContextTimeoutSeconds` to the system config schema and pass it to `ConversationContext` at construction instead of using the hardcoded default.

**Current behavior:**
`ConversationContext` (built in #38) accepts a timeout at construction with a hardcoded default of 30 seconds. `VoiceAutomationService` (wired in #39) constructs contexts but has no config-driven timeout.

**Desired behavior:**
The system config schema gains `conversationContextTimeoutSeconds?: number` (default `30`). `VoiceAutomationService` reads this value from `ConfigRepository` at startup and passes it as `timeoutMs` when constructing each `ConversationContext`. `data/config.json` includes the field with its default value.

**Key interfaces:**
- System config type (wherever the config schema is defined) — add `conversationContextTimeoutSeconds?: number`
- `data/config.json` — add `"conversationContextTimeoutSeconds": 30`
- `VoiceAutomationService` — read `config.get()` at startup (already done for `intentConfidenceThreshold`); extract `conversationContextTimeoutSeconds ?? 30` and pass as `timeoutMs * 1000` to `ConversationContext` construction

**Acceptance criteria:**
- [ ] Config schema type includes `conversationContextTimeoutSeconds?: number`
- [ ] `data/config.json` includes the field with value `30`
- [ ] `VoiceAutomationService` passes the config value (converted to ms) to `ConversationContext` at construction
- [ ] When config value is absent, default of 30s is used
- [ ] Test: constructing the service with a config that specifies a non-default timeout results in contexts that expire at that timeout

**Out of scope:**
