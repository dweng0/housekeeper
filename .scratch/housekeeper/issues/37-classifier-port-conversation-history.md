# Classifier port: add optional conversationHistory

## What to build

Add an optional `conversationHistory` field to the `IntentClassifier` port input type. Update `openai-intent-classifier.ts` to prepend those turns as native OpenAI `user`/`assistant` messages between the system prompt and the current utterance. No behavior change for callers that omit the field.

See ADR 0007.

## Acceptance criteria

- [ ] `IntentClassifier` port input type gains `conversationHistory?: Array<{ role: "user" | "assistant"; content: string }>`
- [ ] `openai-intent-classifier.ts` prepends history turns between system prompt and current user message when field is present
- [ ] Existing callers (no history) pass unchanged — all existing tests still green
- [ ] New unit test: history turns appear in the OpenAI request in correct order

## Agent Brief

**Category:** enhancement
**Summary:** Add optional `conversationHistory` field to `ClassifyOptions` and wire it into the OpenAI classifier as native chat turns.

**Current behavior:**
`ClassifyOptions` (the input type for `IntentClassifier.classify()`) has fields `utterance`, `residentId`, `memories`, and `location`. The OpenAI classifier builds a messages array with a system prompt followed by a single user message containing the utterance. There is no way for callers to pass prior conversation turns.

**Desired behavior:**
`ClassifyOptions` gains an optional `conversationHistory?: Array<{ role: "user" | "assistant"; content: string }>` field. When present and non-empty, the OpenAI classifier prepends these turns as native OpenAI `ChatCompletionMessageParam` objects between the system prompt and the current user utterance. When absent or empty, behavior is identical to today.

**Key interfaces:**
- `ClassifyOptions` — add `conversationHistory?: Array<{ role: "user" | "assistant"; content: string }>` (optional, so all existing callers compile and run unchanged)
- `makeOpenAIIntentClassifier` implementation — when building the messages array, insert history turns after the system prompt and before the final user message
- ADR 0007 documents this decision; do not use Option A (text injection into system prompt)

**Acceptance criteria:**
- [ ] `ClassifyOptions` has `conversationHistory?: Array<{ role: "user" | "assistant"; content: string }>`
- [ ] OpenAI classifier includes history turns in correct position: system prompt → history turns → current user utterance
- [ ] Callers that omit `conversationHistory` see no behavior change — all existing tests pass unmodified
- [ ] New unit test: when `conversationHistory` has N turns, the resulting OpenAI messages array contains system + N history turns + current user, in that order

**Out of scope:**
- Token trimming / budget enforcement (that is `ConversationContext`'s responsibility — #38)
- Any caller actually passing `conversationHistory` (that is #39)
- Changes to the classifier's system prompt or response parsing
