# ListeningWindow mode switching

Status: ready-for-agent
Category: enhancement

## What to build

Add mode-aware filtering to ListeningWindow. Implement "stop-word-only" mode: suppress ambient utterances, match only hard-coded Stop-words via keyword matching.

## Acceptance criteria

- [ ] ListeningWindow has `mode: "normal" | "stop-word-only"` property
- [ ] Add `setMode(mode)` method to public interface
- [ ] Hard-coded stop-word list: "wait", "stop", "hold on", "cancel", "no", "nope", "never mind"
- [ ] In "stop-word-only" mode:
  - [ ] Non-stop utterances logged but not dispatched to `onAmbientUtterance`
  - [ ] Stop-words matched whole-word, case-insensitive via regex: `\b(word)\b`
  - [ ] Matching stop-words dispatched as ambient utterances
- [ ] In "normal" mode: all utterances dispatched (existing behavior)
- [ ] Unit tests cover: mode switching, word boundary matching, false positive prevention
- [ ] Example: "desktop" does NOT match "stop"

## Blocked by

None — can start immediately

## Notes

Stop-word list is hard-coded (not configurable). Per ADR 0009.

## Agent Brief

> *This was generated by AI during triage.*

**What's this about?**

Stop-word interruption requires ListeningWindow to suppress ambient utterances while TTS plays, only matching hard-coded stop-words. This prevents misheard system audio from triggering spurious intents.

**Where does it touch?**

- `src/voice/listening-window.ts` — add mode property, mode-aware filtering

**What's already there?**

- `addUtterance()` already dispatches to `onAmbientUtterance`
- System name detection logic already in place
- Can reuse existing patterns for filtering

**How to verify?**

1. Add `mode: "normal" | "stop-word-only"` property
2. Add `setMode(mode)` public method
3. In "stop-word-only" mode:
   - Build regex: `\b(wait|stop|hold on|cancel|no|nope|never mind)\b` (case-insensitive)
   - Test utterance against regex
   - If match: dispatch to `onAmbientUtterance`
   - If no match: suppress (log debug, don't dispatch)
4. In "normal" mode: dispatch all utterances (existing behavior)
5. Unit tests:
   - Mode switching works
   - "wait" matches, "waiting" doesn't (word boundary)
   - "stop" matches, "desktop" doesn't
   - All 7 keywords work in isolation
   - Case-insensitive: "WAIT", "Wait", "wait" all match
6. No integration with other systems yet; tests use mocks

**Acceptance:** Mode switching works, filtering accurate, tests pass, no regressions
