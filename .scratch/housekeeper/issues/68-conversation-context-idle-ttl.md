# Conversation Context idle timeout

Status: ready-for-agent
Category: bug

## What to build

Reintroduce an idle timeout on `ConversationContext` so the dialogue closes after a period of silence. See ADR 0013.

The previous timer was removed by issue #51 in favour of token-budget bounding alone. In practice this leaves the mic-open dialogue path live forever, so household speech, TV audio, and whisper artefacts (`[BLANK_AUDIO]`, `(sighs)`) flow into the Intent Classifier as Follow-up Utterances. The classifier responds (often in prose), the loop perpetuates.

Implementation must use a **lazy** check on `isOpen()` — no `setTimeout`. ADR 0013 explains the rationale.

## Acceptance criteria

- [ ] `ConversationContext` records `lastTurnAt` on every `addTurn(...)`
- [ ] `isOpen()` returns `false` when `Date.now() - lastTurnAt >= idleTimeoutMs`, even if turns are present
- [ ] `idleTimeoutMs` injected at construction; default `30_000`
- [ ] No `setTimeout` / `setInterval` introduced (lazy read-time check only)
- [ ] `addTurn` after timeout reopens the context (sets `open = true`, refreshes `lastTurnAt`) — opening path is still turn-driven per #51
- [ ] `reset()` behaviour unchanged
- [ ] `voice-automation-service.ts` reads `idleTimeoutMs` from config (alongside `historyTokenBudget`, `intentConfidenceThreshold`) and passes it to `makeConversationContext({ idleTimeoutMs })`
- [ ] Config schema (`ports.ts` and JSON config adapter) gains `conversationIdleTimeoutMs?: number`
- [ ] Unit tests with `vi.useFakeTimers()` cover: open after addTurn, still open just before timeout, closed just after, addTurn after close reopens, reset closes immediately
- [ ] Existing tests still pass
