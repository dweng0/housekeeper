# Bug: ambient utterance response discarded in open conversation context

## What's broken

When a conversation context is open (user received a TTS response to a directed question), a follow-up utterance that doesn't contain the wake word is routed to `onAmbientUtterance`. The handler classifies the utterance ‚Äî and the LLM returns a valid `query` intent with a `response` field ‚Äî but the return value is **silently discarded**. TTS never runs. Conversation context is never updated.

## Reproduction

1. Directed question: "Hey housekeeper, what are your thoughts on travelling to Holland from the UK?" ‚Üí TTS responds correctly.
2. Follow-up (no wake word): "Oh, you can go there via train."
3. Log shows `[Classifier] LLM response: { "type": "query", "response": "Yes, you can take the train..." }` ‚Äî correct intent classified.
4. No `[VoiceAutomation] Classified intent` log entry. No TTS. Response dropped.

## Root cause

`src/voice/voice-automation-service.ts` ‚Äî `onAmbientUtterance` callback (lines 82‚Äì97) calls `await classifier.classify(...)` but ignores the return value. No speech, no context update.

```ts
// current ‚Äî result thrown away
await classifier.classify({ utterance: text, ... });
```

## Fix

Use the classify result. If context is open and intent has a spoken response, speak it and add the turn to context ‚Äî same pattern as `onDirectedQuestion` uses for `query` type.

- [ ] Follow-up utterance in open context (no wake word) triggers TTS when LLM returns a `query` intent with a response
- [ ] Spoken response is added to conversation context history (so subsequent turns have continuity)

- [ ] Unit test: `onAmbientUtterance` with closed context ‚Üí classifier not called
