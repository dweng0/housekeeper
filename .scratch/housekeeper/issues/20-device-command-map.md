# Device command map for payload translation

## What to build

Add `commandMap` to the `Device` type so the voice layer can speak human commands ("on", "off", "lock") while the gateway publishes the correct firmware-specific payload ("1", "0", `{"state":"ON"}` etc).

Closes the gap identified in #19: without this, `DeviceGateway.publish` sends the raw LLM-extracted command string, which may not match what the device firmware expects.

## Acceptance criteria

- [ ] `Device` type includes `commandMap?: Record<string, string>`
- [ ] `VoiceAutomationService` resolves `intent.command` through `commandMap` before publishing (falls back to raw command if no map entry)
- [ ] `DeviceRepository` persists and retrieves `commandMap`
- [ ] Dashboard device detail view shows commandMap entries (key → value pairs)
- [ ] Dashboard allows adding/editing/deleting commandMap entries

## Implementation notes

`commandMap` is optional — devices without it fall back to publishing the raw command string. This keeps #19 shippable before this issue lands.

Example:
```ts
device.commandMap = { on: "1", off: "0", toggle: "2" }
// intent.command = "on" → publish "1"
```

Translation happens in `VoiceAutomationService`, not in `DeviceGateway` — gateway stays a thin MQTT wrapper.

## Out of scope

- Auto-population of commandMap (separate issue)
- Per-automation command overrides

## Blocked by

#19 (device-control intent must be wired before commandMap translation is exercised)
